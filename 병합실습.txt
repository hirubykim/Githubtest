두 버전 사이에 동일한 코드를 수정하면 어떻게 될까요
일단 두 커밋이 서로 같은 코드를 수정했다면 병합커밋을 만들다가 충돌이 날 가능성이 있습니다. 
그래서 동료들과 같이 쓰는 master 브랜치에 바로 병합하지 않고 나만 쓰는 브랜치에 먼저 병합을 한 뒤
병합된 커밋이 문제가 없는 것을 확인하고 master 브랜치에 반영합니다.

물론 master 브랜치에 바로 병합해도 되지만 출돌이 나더라도 master 브랜치에 바로 해결하면 됩니다.
다만 다른 사람들이 불편해지는 상황을 방지하기 위해서 나만 쓰는 브랜치에서 먼저 병합하는 것입니다.
버그가 발생해도 먼저 본인에게 보이는 것이 좋으니깐..


그럼 나만 쓰는 브랜치로 체크아웃 한 후 병합려는 커밋(master 브랜치의 최신커밋)을 클릭하고 병합을 선택합니다.
>> 만약 병합 충돌이 발생하면 서로 다른 내용이 겹쳐있는 경우이므로 내용을 수정해줘야 합니다.

오류 발생 시 "커밋하지 않은 변경사항"에 문제가 되는 파일의 부분을 알 수 있습니다.
해당 파일이 충돌이 되었기 때문에 "스테이지에 올라가지 않은 파일" 목록에 있습니다. 
vscode로 확인을 해보면 git 충돌이 발생한 부분을 자동으로 확인할 수 있습니다. 
'============' 을 기준으로 위 아래에 충돌이 발생한 다른 내용을 알 수 있습니다. (색으로도 구분이 가능하게 표시되어있습니다.)

내용을 파악 후 수정해주고 저장하면 됩니다.
참고로 vscode에는 어떤 내용을 선택할 지 버튼으로 선택하게 기능이 구현되어있습니다. 

내용 수정 후 저장까지 한 뒤 sourcetree에서 확인해보면 충돌이 해결된 것을 알 수 있습니다. 

중요! 이후 다시 병합작업을 실시해야 합니다.
커밋(병합커밋 생성) - 코멘트 작성 (오류 시 자동 생성됨) - push (원격 저장소에 반영함) 
이후 병합된 커밋이 아무 문제가 없다는 것을 확인하고 master 브랜치에도 반영한다.
master 브랜치가 기준이 되어야 하므로 master 브랜치로 체크아웃함
최신 커밋을 선택 후 '병합' 클릭함
master 브랜치의 포인터가 최신 커밋으로 옴
그러나 원격저장소인 origin/master는 최신 커밋이 아니므로 push를 눌러 최신버전으로 반영해줌

브랜치를 합치는 예의바른 방법 = pull request

이제 어떤 상황에서든 브랜치를 합칠 수 있습니다.
그런데 충돌만 해결했다고 무조건 내 브랜치를 master 브랜치에 병합해도 될까요?
master 브랜치는 회사에서 완벽한 코드만 올리자고 약속한 경우가 많습니다. 그러면 내가 이 브랜치에서 무엇을 바꾸었는지 
협력자가 확인할 수 있는 과정을 거쳐야 하지 않을까요? 신입 개발자가 master 브랜치에 코드를 병합하기전 사수 개발자에게 허락을 구하는 겁니다.
이때 pull request가 필요합니다. 풀 리퀘스트는 협력자에게 브랜치 병합을 요청하는 메시지를 보내는 것입니다.
예를 들어 "A브랜치로 B브랜치를 병합해도 되겠니? 수정사항은 다음과 같아"라고 메시지를 보내준다면 보다 수월한 협업이 될 수 있을 겁니다.

깃허브 원격저장소 사이트에 들어가면 방금 push한 브랜치를 알려주면서 "compare&pull request" 버튼이 있습니다. 
"compare&pull request" 버튼은 최근에 푸시한 브랜치가 있을 때만 보여집니다. 다른 브랜치로 풀리퀘스트를 보내고 싶거나 직접 설정을 하고 싶다면
그 아래 줄에 있는 "new pull request" 버튼을 누르면 됩니다.

풀리퀘스트 버튼을 누르면 메시지도 함께 적을 수 있는 화면도 나옵니다.
여기서 중요한 것은 베이스 브랜치와 비교 브랜치입니다. 
병합할 때 병합한 결과물을 두 브랜치 중 어느 곳에 올려야 하는지 정해야 한다고 했습니다. 
여기서 병합 결과물이 올라갈, 즉 기준이 되는 브랜치가 베이스 브랜치입니다.
현재 기준의 브랜치와 비교 대상이 되는 브랜치가 비교 브랜치입니다. 
"compare&pull request" 버튼을 누르면 자동으로 세팅을 해주지만 직접 설정해서 pull request를 보낼 때는 직접 설정을 해야 합니다.

베이스 브랜치와 비교 브랜치 옆에 "able to merge" 문구가 있습니다. 두 브랜치가 충돌 없이 병합될 수 있음을 알려주는 문구입니다.
만약 충돌이 발생한다면 빨간색 문구로 "conflict"가 뜹니다.

왼쪽 사이드 바를 보면 reviewers, assignees, labels 항목이 보입니다.
차례대로 reviewers는 검토 요청을 받는 사람입니다.
저장소의 협력자가 여려명이라면 여러명에게 검토 요청을 할 수 있습니다. 보통 같은 팀원이나 해당 기능과 연관된 동료를 선택합니다.

그 다음 assignees는 이 풀 리퀘스트를 담당하는 동료입니다.
보통 자기 자신입니다.

마지막으로 labels는 이 풀 리퀘스트에 관한 라벨을 달아줍니다. 
보통 버그, 리뷰 필요, 프론트엔드, 백엔트 등이 있습니다.

풀 리퀘스트를 보내고 요청도 수락된다면 깃허브 원격저장소 사이트에 pull request 항목에서 closed 항목에 보낸 요청이 들어갈 것입니다.

병합이 잘 되었는 지 sourcetree에서 확인을 해보면 간혹 원격 저장소의 origin/master 브랜치가 옛날 커밋을 가리킬 수 있습니다.
이럴 경우 패치를 해야 합니다. git에서 새로운 이력을 업데이트하는 명령어가 패치입니다. 
풀이 실제 코드를 내려받는데 비해 패치는 그래프만 업데이트 합니다. 소스트리의 경우 10분 단위로 자동으로 패치를 합니다.
즉각적으로 병합된 내용이 그래프에 반영되지 않는다면 수동을 패치하면 됩니다.

풀 리퀘스트를 보내고 수락 받은 이후에도 내 컴퓨터의 master 브랜치에 새로운 커밋을 반영하기 위해 따로 pull 해줍니다.

내 컴퓨터의 master와 원격저장소의 origin/master가 동일한 커밋을 가리키고 있으면 정상적으로 처리된 것입니다.
