만약에 풀 리퀘스트를 보냈는데 코드 충돌이 발생한 상황이라면 어떻게 해결할 것인가?
단순히 내 브랜치로 먼저 병합하여 충돌을 해결한 뒤 다시 풀리퀘스트를 보내면 충돌이 발생하지 않을 것입니다.
하지만 그러면 풀 리퀘스트에 내가 추가한 코드 이외에도 충돌을 해결하느라 생긴 병합 커밋이 생길 것 입니다.
이러한 상황을 피하고 깔끔하게 변경한 부분만 풀 리퀘스트를 보낼 수 있는 방법에 대해 소개하겠습니다.


1.원본 저장소에 새로운 커밋이 있는데 포크한 내 원격 저장소에는 커밋이 안보이는 경우
만약에 원본 저장소의 소유자와 포크한 원격 저장소의 소유자가 같은 코드를 고쳐 충돌이 발생한 상황이 있다고 가정해봅니다.
첫번째로 포크한 원격 저장소의 소유자가 코드를 수정하고 push 합니다. 
두번째로 원본 저장소의 소유자가 최신 업데이트를 받아옵니다. (최신 상태로 반영되지 않은 경우 소스트리 master 브랜치 옆에 숫자가 표기됨. master 브랜치 클릭 후 "가져오기 origin/master (추적됨)" 클릭)
이렇게 해서 원본 저장소에서 최신 업데이트를 pull해서 받아옵니다. 숫자가 사라지면 정상적으로 pull 된 것이고 원본 저장소에서 더 이상 받아올 커밋이 없는 최신 상태라는 의미입니다.

이제 충돌이 나는 상황을 만들기 위해 원본저장소 소유자가 본인 로컬 저장소에 있는 파일을 수정합니다.
이는 포크한 원격 소유자의 로컬 저장소에 있는 파일과는 다릅니다.

원본 소유자와 포크한 원격 소유자가 서로 같은 코드를 수정했습니다.

포크한 원격 소유자의 경우 원본 소유자가 서로 같은 코드를 고쳤다라는 사실을 모릅니다.
이때 만약 포크한 원격 소유자가 풀 리퀘스트를 보내면 충돌이 발생합니다.

원본 저장소의 소유자가 내용을 변경하고 만약 포크한 원격 소유자가 해당 사실을 모른 채로 풀 리퀘스트를 보낸다고 가정합시다.
이때 포크한 원격 소유자가 풀 리퀘스트를 보내려고 New pull request 버튼을 클릭하면 빨간색 글씨로 Can't automatically merge 라는 메시지가 뜹니다.
코드가 충돌났다는 것을 말해주는 메시지입니다.
이렇게 되면 충돌을 해결하고 다시 풀 리퀘스트를 보내야 합니다.


2.여러 원격저장소 히스토리를 한눈에 보기: 리모트 추가
지금까지 상황을 정리하자면 포크한 원격 저장소 소유자가 원본 저장소를 포크하면서 새로운 원격저장소를 만들었고 포크한 시점까지의 모든 히스토리는 포크한 원격 저장소의 소유자가 알 수 있지만 
그 다음에 원본 저장소에서는 무슨 일이 일어나는 지 모른다는 것입니다.

원본 저장소와 포크한 원격 저장소는 이미 주소까지 바뀌어서 서로 다른 원격 저장소가 되어버렸기 때문입니다. 
그럼 포크한 사용자 입장에서 자신의 원격 저장소에서 다른 원본 저장소의 히스토리도 보고 싶다면 어떻게 해야 할까요?

방법은 소스트리가 원본저장소와 원격저장소를 동시에 추적하도록 만들면 됩니다.

참고: "이미 소스트리에서 원본 저장소와 원격 저장소를 동시에 추적하고 있지 않나? 원본 저장소의 탭과 원격 저장소의 탭이 보이잖아" 라고 생각할 수 있는데 
이는 조금 다른 이야기 입니다. 소스트리에서 여러 로컬 저장소를 관리하고 있지만 각 로컬 저장소에서는 하나의 원격 저장소를 바라보고 있습니다. 
지금 할 실습은 하나의 로컬 저장소에서 두개 이상의 원격 저장소를 바라보는 방법을 다룰 것입니다.

포크한 원격 저장소의 소유자 계정으로 들어갑니다. 
상단에 있는 "저장소" -> "원격 저장소 추가"를 클릭합니다. 

(원격 저장소의 소유자 계정은 원격저장소만 origin이라는 이름으로 바라보고 있어 이제 원본저장소 주소를 다른 이름으로 추가하는 과정입니다)
(소스트리 왼쪽 사이드바를 보면 원격탭에 origin만 있는데 이곳에 원본 저장소를 추가하면 추가한 원본 저장소가 하나 더 생길 것입니다)

내 컴퓨터의 로컬 저장송와 연결된 원격저장소의 경로가 보입니다. 
보면 포크했던 원격저장소 주소가 origin이라는 이름으로 저장되어 있습니다.
아래에 추가 버튼을 눌러 추가해줍니다. (새롭게 추적하고 싶은 원본저장소는 새로운 이름으로 추가해줍니다)

참고: 맨 처음에 git remote add origin [원격저장소 주소] 명령어를 사용한 기억이 있을 것입니다.
해당 명령어는 원격저장소의 주소를 origin 이라는 이름으로 저장하라는 의미입니다. (원격저장소에서 클론할 때 이 origin이라는 이름으로 원격저장소가 자동으로 저장됩니다)
소스트리에서 origin/master 와 master 차이점을 볼 때 origin/master가 원격저장소에 올라간 브랜치라는 것을 보면 원격저장소의 닉네임이 origin이라는 것을 알 수 있습니다.

새로운 원격저장소의 이름을 upstream 이라고 지어줍니다.(편의상 다른 걸 사용해도 됩니다)
하지만 이것은 원본 저장소를 가리키는 관용적 닉네임입니다. "url/경로"에는 원본 저장소의 주소를 복사해서 넣어줍니다.
확인을 누르면 upstream이라는 닉네임으로 원본 저장소가 추가된 것을 볼 수 있습니다.

돌아와서 왼쪽 사이드바에 원격탭을 보면 origin 외에도 upstream이 추가된 것을 볼 수 있습니다.

upstream을 클릭하고 "upstream에서 가져오기" 항목을 선택합니다. 이것은 upstream의 원본 저장소에 있는 커밋 히스토리를 받아오는 것 입니다. 
이를 패치(fetch)라고 합니다.

패치는 간단하게 git의 새로고침 기능입니다. 패치를 하면 원본저장소의 이력을 업데이트합니다. 
예를 들어 동료 개발자가 master 브랜치에 커밋 5개를 올렸는데 내 소스트리에서는 아직 그 이력이 안보일 때 패치를 하면 이력이 보입니다.
이력만 가져오는 거니까 내 코드에는 아무 영향이 없습니다. 최신 코드를 내 코드에 반영하는 pull과 다릅니다.

참고: 소스트리가 10분에 한번씩 자동으로 패치를 해주기 때문에 따로 패치하지 않아도 새로운 이력이 보입니다. 소스트리의 편의 기능입니다.

이제 그래프를 보면 origin/master에는 포크한 원격저장소 소유자의 코드가 upstream/master에는 원본저장소의 소유자 코드가 있을 거란 걸 알 수 있습니다.
포크한 원격저장소의 소유자가 코드를 수정했지만 너무 예전 코드를 기점으로 코드를 수정했다는 것을 이제 알 수 있습니다.
이렇게 하면 원본저장소의 최신 코드와 충돌이 날 가능성이 있습니다.

3.묵은 커밋을 방금 한 커밋처럼: 리베이스
풀 리퀘스트를 보냈을 때 충돌이 난다면 우리는 2가지 방법을 선택할 수 있습니다.

첫번째는 현재 커밋과 병합하고 싶은 커밋을 미리 내 브랜치에 병합해서 병합커밋을 만들고 이것을 풀리퀘스트를 보내는 방법입니다.
예를 들어 포크한 원격저장소의 1번 커밋과 원본 저장소의 2번 커밋을 합치려고 하니까 충돌이 일어났습니다.
포크한 사용자는 1과 2의 충돌을 해결한 3번 커밋(병합커밋)을 자신의 원겨저장소에 올렸습니다. 
그러면 이미 포크한 원격저장소에서 충돌이 해결된 상태이므로 2번 커밋과 충돌없이 병합할 수 있을 것입니다. 
그러나 이렇게 먼저 병합하고 풀 리퀘스트를 보내면 충돌은 나지 않겠지만 포크한 사용자 입장에서 불필요한 병합커밋의 이력을 남기게 됩니다.

두번째는 바로 묵은 커밋을 방금 한 커밋처럼 이력을 조작하는 것입니다. 
이 묵은 커밋을 방금 한 커밋으로 바꾸는 방법은 첫번째 방법보다 깔끔하고 쉽습니다. 

예를 들어 포크한 사용자가 0번 커밋을 베이스로 1번과 2번 커밋을 만들었습니다. 
0번 커밋이 옛날 커밋이어서 최신 커밋인 3번 이랑 병합하려면 충돌이 납니다.
그런데 1번, 2번 커밋을 0번 커밋이 아니라 3번 커밋을 베이스로 만들면 문제가 발생하지 않습니다.
즉, 커밋의 베이스를 떼어내서 다른 곳으로 붙이는 것을 리베이스(rebase)라고 합니다. 
병합 커밋이 생기는 첫번째 방법과 달리 이력이 깔끔하게 남습니다. 그리고 내 브랜치에 내 변경사항만 남길 수 있다는 장점도 가지고 있습니다.

       /------------3
-----0 ------1 -----2

       /------------3 ------1 -----2
-----0 

간단히 정리하자면 이전 코드를 기준으로 만들었던 브랜치를 마치 최신 코드를 기준으로 만든 것처럼 이력을 조작하는 것입니다.
만약에 충돌이 있다면 이력을 조작하는 중간에 조용히 고쳐주면 됩니다.

리베이스하는 방법
새로운 베이스를 삼고 싶은 커밋을 클릭하고 마우스 오른쪽 버튼을 눌러 "재배치"를 클릭합니다. 
경고 창이 나오면 확인 버튼을 눌러줍니다. 이는 매우 의미있는 경고창입니다. 리베이스를 하려면 다른 개발자가 이 변경사항을 사용하고 있지 않아야 한다는 말입니다.
현재 있는 포크한 원격저장소의 MASTER 브랜치를 본인 혼자만 사용하는 것이냐고 묻는 경고창입니다. 
리베이스는 히스토리를 강제로 조작하기 때문에 다른 사람이 만약에 이 히스토리를 보고 있다면 완전히 꼬이고 말것입니다. 
그렇기 때문에 리베이스는 반드시 혼자만 쓰는 브랜치에서 수행해야 합니다. 보통 기능 구현을 하는 브랜치는 혼자만 쓰는 것이기에 편하게 리베이스해도 됩니다.

다음 경고창으로 충돌 경고창이 나오는데 실무에서 협업할 때에는 충돌이 비일비재하므로 충돌이 발생하면 고쳐주면 됩니다.

이후 커밋하지 않은 변경사항이라는 문장이 보이고 충돌이 난 파일이 스테이지 위와 아래에 동시에 고쳐진 것을 확인할 수 있습니다.
눌러보면 익숙한 충돌 메시지가 보이고 화살표 괄호로 같은 부분이 고쳐진 코드를 구분해서 보여주고 있습니다.
비쥬얼 스튜디오 코드에서 해당 파일을 열어서 충돌을 해결해주고 저장합니다.
충돌을 해결한 파일을 스테이지에 올립니다.
소스트리에서 "액션-재배치 계속" 을 클릭해서 리베이스를 계속 진행합니다. 
리베이스는 커밋을 하나씩 비교하여 충돌이 있는지 없는지 확인하기 때문에 계속 같은 곳을 수정했다면 [재배치 계속] 버튼을 누를 때마다 충돌이 여러번 일어날 것입니다.

모든 충돌이 해결되고 리베이스가 완료되었습니다.
베이스 커밋이 변경된 것을 확인할 수 있지만 내 로컬저장소 브랜치인 master는 제대로 리베이스가 된 것처럼 보이는데 원격저장소 브랜치인 origin/master는 아직도 옛날 커밋을 베이스로 삼고 있습니다.
이제 push를 해서 로컬 저장소의 이력을 원격저장소에 반영해야 합니다. 
다만 여기서 주의할 점이 있습니다. 리베이스는 이력을 조작하는 행위이기 때문에 일반 푸시로는 수행할 수 없습니다.

리베이스를 하려면 다른 개발자가 이 변경사항을 사용하고 있지 않아야 한다는 말입니다. 리베이스는 히스토리를 강제로 조작하기 때문에 다른 사람이 만약 이 히스토리를 보고 있다면 완전히 꼬여버립니다.
이력을 조작하고 푸시하는 것은 다른 개발자에게 위험한 행위여서 무조건 혼자 사용하는 브랜치에서만 해야 합니다. 그리고 이렇게 이력을 조작하면 특수한 옵션을 붙여서 푸시해야 합니다. 
바로 [강제 푸시] 입니다.

[도구 - 옵션]을 클릭합니다. git탭에서 "강제 푸시 기능"을 체크합니다. 
이렇게 해야 push할 때 강제 푸시 체크박스를 볼 수 있습니다.
다시 push해줍니다. 강제 푸시 체크박스가 생긴 걸 볼 수 있고 이러면 히스토리가 조작된 커밋도 문제 없이 원격저장소에 올라갑니다.
강제 푸시 경고창이 나오면 "예" 를 눌러줍니다.
이후 원격저장소에서도 리베이스된 모습을 볼 수 있습니다.

충돌을 해결했으니 다시 한번 풀리퀘스트를 보냅니다.
Can't automatically merge 가 아닌 able to merge 메시지가 뜹니다. 
풀리퀘스트를 보내고 소유자가 승인해주면 됩니다.
